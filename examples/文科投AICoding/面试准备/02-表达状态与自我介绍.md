## 通用自我介绍（45 秒口语版）

> 您好，我是一个偏产品原型和 AI coding 应用方向的候选人。过去我最完整的一段项目是 EventEase：因为校园活动里报名、签到和统计都很分散，我用 AI coding 做了一个轻量 Web demo，把报名表单、二维码签到和出勤导出串起来。这个项目里，代码实现大量用了 AI 辅助，但产品流程、字段设计、边界场景和验收标准是我自己拆的。我想投这个方向，是因为我比较擅长把真实业务问题拆成能跑起来的原型，再用用户反馈继续迭代。

## 关键字卡

不要照读上面那段。面试前只看这张卡：

- 真实痛点：报名分散、签到靠人工、统计麻烦
- EventEase：报名、签到、导出
- 我负责：流程、字段、边界、验收
- AI 辅助：实现提速，不冒充工程深度
- 岗位匹配：把业务问题变成可验证 demo

## 项目逐字稿 · EventEase（90 秒）

| 模块 | 该怎么讲 |
|------|----------|
| 背景 | 校园活动报名信息散在表格/群消息里，签到靠人工点名，活动后统计出勤很耗时间。 |
| 我做了什么 | 我把这个流程拆成报名、签到、导出三个核心环节，用 AI coding 做出一个可运行 demo。 |
| 分步怎么做 | 先画页面流，再定 event / participant / signup status / check-in time 字段，之后做报名表单、二维码签到和导出。 |
| 结果 | demo 能跑通小型活动从报名到签到统计的闭环；测试时根据反馈补了提示文案和异常补签说明。 |
| 困难 | 最大难点不是页面，而是哪些信息必须收集、哪些不该收集，以及异常签到怎么处理。 |
| 下次优化 | 先补组织者权限、重复签到限制和活动后复盘视图，再考虑更多活动模板。 |

## 不会就不会

| 被问到 | 安全回答 |
|--------|----------|
| 你能不能手写完整后端？ | 我现在不能把自己包装成独立后端工程师。这个项目里我能讲清产品拆解、字段、流程和验收，代码实现主要是 AI 辅助完成。 |
| 这个技术方案是不是最优？ | 我不敢说最优。它适合小型活动 demo。如果要上线，我会优先验证权限、隐私和异常处理，再让工程同学评估架构。 |
| 你有没有大规模用户数据？ | 没有大规模上线数据。这里更准确的说法是小范围 demo 测试和反馈，不会夸大成正式产品数据。 |
